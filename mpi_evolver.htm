概述

MPI（Message Passing Interface，消息传递接口）是一种用于在多个进程之间传递消息的协议，这些进程通常运行在不同的机器上。MPI 版本的 Surface Evolver 可以在多台机器的多个处理器上执行 Evolver，所有处理器处理同一个表面，由一台机器控制其他机器。假设用户已熟悉 MPI，并且已安装 MPI。

MPI Evolver 仍处于早期开发阶段，尚不具备生产使用所需的功能。特别是，它不能安全地进行拓扑变化（topology changes）。目前使用 MPI Evolver 的人都只是因为喜欢尝试新工具。

MPI Evolver 的运行架构是将任务 0（task 0）作为主任务（master task），通过命令行界面与用户交互，同时管理一组从任务（slave tasks）。每个从任务都是 Evolver 的完整版本，只是它从主任务接收命令，并在关键点与其它任务同步数据。每个从任务拥有整个表面的一部分，但主任务不拥有表面数据。顶点（vertices）、边（edges）和面片（facets）分配在各从任务之间，但所有任务（包括主任务）都知道所有的体（bodies）（至少目前如此）。

每个任务上的表面被分为属于该任务的"本地"（native）元素和从其他任务复制而来的"边缘"（corona）元素。目前实现了三个级别的边缘状态：

  • 级别 0：没有边缘元素。这是数据文件读入后的初始状态，但通常用户不会看到此状态。
  • 级别 1：本地面片的边缘顶点和边已存在。这足以进行普通表面张力演化的计算，是加载后用户看到的默认状态。
  • 级别 2：所有与本地顶点相邻的边缘边和面片及其顶点均已存在。某些能量（如平方平均曲率 squared mean curvature）需要此状态，因为这些能量的计算需要顶点周围更广泛的信息。在 OpenGL 图形显示中，按 'y' 键可切换级别 2 边缘的显示。

编译

除 metis.c 外的所有常规 Evolver 文件，以及 mpi*.c 文件（目前不包括 mpi_sparse.c），应在编译器命令中定义以下宏常量： 
MPI_EVOLVER
TASK_ID_BITS=22
LONG_ID

生成的目标文件应与相应的 MPI 库链接。主任务和从任务使用相同的可执行文件。注意：在 variable.c 中可能出现的变量 nproc 和 procs_requested 与 MPI 毫无关系；请勿修改它们。

如果需要在任务之间方便地重新分区表面，建议链接 PARMETIS 和 METIS 库，可在此处获取。此时，编译时还应定义以下宏常量： 

METIS
PARMETIS

数据文件

有几种方式设置数据文件：
  1. 为每个任务使用相同的常规 Evolver 数据文件。此时整个表面将分配给任务 1（第一个从任务），但所有任务读取相同的头部信息。可以使用下文描述的 "repartition" 命令在从任务之间重新分配表面。
  2. 将所有表面放在一个数据文件中，但为各元素标记其分配到的任务编号。标记方法是在每个元素编号后附加 "@n"，其中 n 是任务编号。这种数据文件的一个优点是它可以被常规 Surface Evolver 读取，后者会忽略 "@n" 标记。例如： 
    Vertices
1@4    0.0000000000    0.0000000000    1.0000000000  fixed
2@1    2.0000000000    0.0000000000    0.0000000000  fixed
3@2    2.0000000000    2.0000000000    1.0000000000  fixed
4@3    0.0000000000    2.0000000000    0.0000000000  fixed
...
Edges
1@4   1@4 546@4  fixed
2@1   2@1 547@1  fixed
3@2   3@2 548@2  fixed
4@3   4@3 549@3  fixed
...
Faces
1@4   -3139@4 -3137@4 -3138@4
2@1   -3142@1 -3140@1 -3141@1
3@2   -3145@2 -3143@2 -3144@2
4@3   -3148@3 -3146@3 -3147@3
...
Bodies
1@1  362@3 384@3 383@3 382@2 381@3 ...
  3. 为每个任务使用不同的数据文件。数据文件结构与方法 2 相同，但省略了未分配给特定任务的元素。在不同任务上使用相同的元素编号是合法的；也就是说，顶点 23@2 与顶点 23@4 是完全不同的元素。每个数据文件应有相同的头部信息。
无论采用哪种方式，只有主任务读取数据文件底部 "read" 部分的内容。从任务不读取 "read" 部分。这样设计是为了让主任务拥有完全的控制权，且所有任务可以读取同一个文件。如果从任务需要进行初始化，主任务应使用下文描述的方法之一来指示。

调用方式

无论以何种方式调用 MPI 任务，每个任务（主任务和从任务）都应在命令行上指定一个数据文件名。数据文件可以各任务不同，也可以相同，如前所述。如果数据文件名包含 "%d"，则该占位符将自动被任务编号替换以形成实际的文件名（实际上可以使用 C printf %d 格式的任何变体，例如 %03d 以确保任务编号为 3 位数字）。这使得单条 mpi 命令行可以在不同任务上加载不同的数据文件。每个任务的数据文件必须可从其运行所在的机器上访问。主任务（任务 0）应在作为控制台使用的机器上运行。

命令执行

所有用户命令都在主任务中输入并由主任务执行。部分命令（如下所列）已被修改为以协调的方式在所有任务上并行执行。其余命令仅在主任务上执行。

变量在每个任务上独立存在；它们不会自动同步。

MPI 版本的特殊命令：

task_exec n,string

让任务 n 将字符串作为命令执行。任何聚合命令（aggregate commands）将在所有本地元素和导入元素上执行。

parallel_exec string

让每个从任务执行该字符串。任何聚合命令将在所有本地元素和导入元素上执行。

repartition

根据各元素属性中设置的任务编号重新分配表面的元素。这些属性声明如下： 

define vertex attribute v_newpart integer[2]
define edge   attribute e_newpart integer[2]
define facet  attribute f_newpart integer[2]
define facetedge attribute fe_newpart integer[2]
define body  attribute b_newpart integer[2]
建议将上述声明行包含在任何 MPI Evolver 数据文件中。用户设置每个属性的第一个分量；第二个分量供内部使用。示例见下文。

metis_readjust n

执行重新分区，但使用 METIS 分区算法，旨在基于当前分区而非从头重新分区。n 为期望的分区数；通常为 mpi_maxtask。

mpi_debug

切换 MPI 调用跟踪的打印。请勿使用。

修改过的常规命令：

这些命令照常在主任务上输入，并作用于所有从任务。聚合命令（如 histogram）仅包含每个任务上的本地元素，不包括导入元素。

foreach

当从主任务作为最外层循环命令运行时，循环体将在每个任务上运行。但需注意：循环体以编译形式传递给从任务，如果内部数据结构（如符号表）与主任务上不完全一致，命令将产生不可预测的结果。这就是为什么在每个任务读取的数据文件中保持完全相同的顶部部分是一个好主意。如果对 "foreach" 是否能在任务间正常工作有疑虑，请使用下文描述的 "parallel_exec"。

从主任务运行的 "foreach" 将在每个任务的本地元素上执行其循环体，但在单个任务上运行的 "foreach"（例如通过 task_exec 或 parallel_exec）将在本地元素和导入元素上执行。

load

加载新的数据文件。数据文件名将分发给各任务；如果包含 "%d"，将被任务编号替换，以便不同任务可以加载不同的数据文件。

r

正常工作。仅在主节点上执行 "r"；它将负责在所有节点间同步数据。请勿在单个任务上执行 "r"。

g

正常工作。在所有任务间同步力和移动。共轭梯度法（conjugate gradient）也能正常工作。

hessian,hessian_seek,eigenprobe,ritz

Hessian 命令正常工作。 默认情况下，hessian 会收集到主任务。如需分布式计算（实验性），请使用 hessian_menu 命令 'A'。

c

从主任务执行时，打印各元素的累积计数；各任务的本地计数会被求和。在单个任务上执行时，c 报告本地元素和导入元素的总和。

V

正常工作。在各任务间同步，因此其效果如同整个表面在一个任务上一样。

define attribute

元素属性定义将传播到所有任务，以保持各任务同步。在各任务上分别定义属性是非常糟糕的做法。

t,l,w,refine,delete,u

请勿尝试这些或其他本地拓扑变化命令（除非您确信其效果完全在一个任务范围内）。这些命令尚未被修改以正确工作。

MPI 版本的内部变量：
this_task

当前任务的任务编号

mpi_maxtask

最高任务编号

corona_state

当前边缘级别，取值 0、1 或 2。 设置此变量可改变边缘状态。将其设置为 0 是非常糟糕的做法。

MPI 版本的元素属性：
mpi_task

此元素所属的任务编号。例如，如果想查看任务 5 从任务 2 导入了哪些顶点，可以使用： 

task_exec 5,"list vertex where mpi_task == 2"

图形显示

获取屏幕图形的两种方式：
  1. 在任务上使用常规屏幕图形。显示将在任务执行所在的机器上进行。当在同一台机器上执行少量任务进行测试时最为有用。使用 "showq" 命令可避免进入图形提示符。例如： parallel_exec "showq" 查看所有部分。
  2. 在主任务上使用屏幕图形，并让其从其他任务导入数据。目前仅在 OpenGL 图形中测试过此方式。 首先，在主任务中用 's' 启动图形显示。默认将显示任务 1 的表面部分。 要显示的任务可在图形显示中选择：按 'M' 进入菜单模式，然后用鼠标右键显示主菜单，接着进入底部附近的 MPI 任务子菜单，选择所需的任务。
如果存在粗边缘（thick corona），按 'y' 键可切换其显示。

示例

假设您有一个标准的 Evolver 数据文件，需要加载并在各任务之间分区。当将其加载到 MPI Evolver 时，表面将完全加载到任务 1 中。然后可以重新分配。如果已链接 PARMETIS 库，可以使用以下脚本： 
// mpi_equalize.cmd
// Spread surface out over all tasks.

define vertex attribute v_newpart integer[2]
define edge   attribute e_newpart integer[2]
define facet  attribute f_newpart integer[2]
define facetedge attribute fe_newpart integer[2]
define body  attribute b_newpart integer[2]

mpi_equalize := {
// set new partition numbers
metis mpi_maxtask;
parallel_exec "set vertex v_newpart[1] vpart+1";
parallel_exec "set edge e_newpart[1] epart+1";
parallel_exec "set facet f_newpart[1] fpart+1";
parallel_exec "set body b_newpart[1] 0";
parallel_exec "set facetedge fe fe_newpart[1] fe.facet[1].fpart";

// do it
repartition;
}
如果没有 PARMETIS，则需要自行设计分区方式，例如基于几何形状。以下脚本将表面分为四个部分： 
define vertex attribute v_newpart integer[2]
define edge   attribute e_newpart integer[2]
define facet  attribute f_newpart integer[2]
define facetedge attribute fe_newpart integer[2]
define body  attribute b_newpart integer[2]

mpi_equalize := {
// set new partition numbers
xmid := avg(vertex,x);  // avg over all tasks, but only master task
ymid := avg(vertex,y);  // knows xmid and ymid
parallel_exec sprintf
    "set vertex v_newpart[1] ((x < %f) ? 1:0)+((y < %f)?2:0)",xmid,ymid;
parallel_exec "set edge ee e_newpart[1] min(ee.vertex,v_newpart[1])";
parallel_exec "set facet ff f_newpart[1] min(ff.vertex,v_newpart[1])";
parallel_exec "set body bb b_newpart[1] 0";  // doesn't matter for bodies
parallel_exec "set facetedge fe fe_newpart[1] fe.facet[1].fpart";
    // facetedge should agree with facet

// do it
repartition;
}
